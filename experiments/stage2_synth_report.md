# 阶段2 指令合成实验报告

## 实验设置

- **API**：DeepSeek v4flash (deepseek-chat) / v4pro (deepseek-reasoner)
- **种子来源**：阶段1 cleaned.jsonl (3082条)，经 SeedCurator 筛选 80 条
- **日期**：2026-06-14 ~ 2026-06-15
- **总费用**：¥2.20

## 实验流程

```
cleaned.jsonl (3082条)
  → SeedCurator 筛选 (长度100-500, PPL<15, 中文>0.5)
  → 80条种子
  → Self-Instruct (v4flash, temp=0.8, 5条/种子)
  → 379条有效指令
  → PostFilter (规则+Jaccard去重)
  → 375条
  → Evol-Instruct (v4pro, temp=0.3, 150条候选)
  → 130条进化指令
  → 合并 → 505条 → synthesized.jsonl
  → DPO (代码20+数学20) → 40对 → dpo_pairs.jsonl
```

## 关键参数选择

| 参数 | 值 | 理由 |
|------|-----|------|
| 种子长度 | 100~500字 | 太短信息不够，太长超 context |
| 种子 PPL | < 15 | 确保种子流畅度 |
| 种子中文占比 | > 0.5 | 排除中英过度混杂 |
| 最大种子数 | 80 | 平衡覆盖与 API 费用 |
| Self-Instruct model | v4flash | 批量生成，需高吞吐低延迟 |
| Self-Instruct temp | 0.8 | 需要多样性，不用过于精确 |
| Evol-Instruct model | v4pro | 深度进化，需要推理能力 |
| Evol-Instruct temp | 0.3 | 进化需精确遵循约束 |
| DPO 代码执行超时 | 5s | 防止死循环，及时丢弃 |

## DPO 偏好数据构造结果

| 指标 | 值 |
|------|-----|
| 目标 | 40 对 (代码 20 + 数学 20) |
| 实际产出 | **40 对** ✅ |

| API 调用 | 63 次 |
| 费用 | ¥0.05 |

### 代码执行判定 — 几乎全军覆没

```
预期：LLM 生成"正确A + 含bugB" → 执行A通过、B不通过 → 自动判定
现实：代码执行成功率仅 5%（1/20）
      → 其余全部走了 LLM-as-Judge 兜底
```

根本原因：LLM 很难生成"刚好语法正确但执行 crash"的代码。生成的 B 要么也通过（bug 太隐蔽），要么语法错直接 parse error（A 和 B 都失败）。

**解决方案**：引入 LLM-as-Judge（v4pro）兜底 → 代码偏好对从 3 对增长到 20 对，成功率 100%。

### 数学答案验证 — 最可靠

成功率 95%，正则 `答案[：:]\s*([\d.]+)\s*$` 匹配稳定。

**局限**：只能验证"答案数值不同"，无法判断哪个是标准正确答案。生产中需题目+标准答案的校验。

### LLM-as-Judge 踩坑

- **max_tokens 陷阱**：v4pro 推理链吃掉 token 预算，初次设置 max_tokens=10 → 解析全失败。改 500 后正常。
- **输出格式兼容**：需兼容 "A"、"回答 A"、"A。"、"我认为A更好" 等多种格式。

## 踩坑记录

| 坑 | 现象 | 根因 | 修复 |
|----|------|------|------|
| VPN 断开 | 231 条后全失败 | 网络连接丢失 | 断点续跑 + http_proxy 环境变量支持 |
| DPO 代码执行全 PASS | 20 对只有 1 对可执行判定 | LLM 难生成"刚好执行 crash"的代码 | LLM-as-Judge (v4pro) 兜底 |
| LLM-Judge 解析失败 | v4pro 输出被截断 | max_tokens=10 被推理链吃光 | 改 500 |
| LLM-Judge 输出格式多样 | "A""回答A""A。"不统一 | v4pro 自由输出 | 兼容多种格式 |
