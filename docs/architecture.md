# AquilaLM知识框架

---



### 一、项目总览：一句话定位



> **AquilaLM 是一个以数据闭环为核心驱动的轻量级大模型训练栈，聚焦于“后训练阶段”（Post\-Training）的数据工程与对齐，通过构建“数据清洗→指令合成→质量评估→课程学习→下游评估反馈”的自进化循环，用数据质量的持续提升带动模型能力的增长。**
> 
> 



**面试话术模板**：

> “这个项目的出发点是：模型能力的上限由数据质量决定。我不想只做‘调参侠’，所以从MiniMind框架出发，自己搭建了一套完整的数据飞轮系统。它的核心逻辑是——让下游模型的评估结果反向驱动上游的数据合成与过滤策略，形成一个能自我迭代的数据闭环。”
> 
>

### 二、项目技术架构全景图



下面用一张“数据流”的逻辑图，帮你把简历里散落的技术点串起来（面试时可以边画边讲）：



```Plain Text
┌─────────────────────────────────────────────┐
                  │           阶段四：下游评估与反馈                │
                  │  · 模型评估（Loss / 评测分数）                │
                  │  · 反馈信号：驱动合成策略调整（温度/采样比）    │
                  │  · 反馈信号：驱动过滤阈值优化                  │
                  └───────────────────┬─────────────────────────┘
                                      │ (反向驱动)
┌─────────────────────────────────────┼──────────────────────────┐
│            阶段三：课程学习 & 模型训练                          │
│   · 从易到难排序 → 送进训练                                      │
│   · SFT / DPO / PPO（RLHF）                                     │
└─────────────────────────────────────┼──────────────────────────┘
                                      │
┌─────────────────────────────────────┼──────────────────────────┐
│            阶段二：指令合成 & 偏好数据构造                       │
│   · Self-Instruct / Evol-Instruct                               │
│   · DPO 偏好数据自动构造（规则+RM）                             │
└─────────────────────────────────────┼──────────────────────────┘
                                      │
┌─────────────────────────────────────┼──────────────────────────┐
│            阶段一：数据清洗                                        │
│   · 规则过滤 → KenLM 困惑度初筛 → MinHash+LSH 去重             │
└─────────────────────────────────────┴──────────────────────────┘
```

**这张图的逻辑是**：一条数据要经过“清洗→合成→评估→排序→训练→评估”六个环节，而最后一环的“下游评估结果”会作为反馈信号，回到“指令合成”和“数据过滤”阶段，调整参数——这就是“数据飞轮”。

### 三、分模块详解：定位、内容、面试核心问题



下面逐个模块，用“是什么→为什么→怎么做的”三层结构拆解，并附上面试中可能会被追问的核心点。



#### 模块1：数据清洗流水线

**定位**：飞轮的“入口过滤器”，保证进入后续环节的原始种子数据是干净、去重的。



**内容拆解**（简历中的原文）：

> 规则过滤 → KenLM 困惑度初筛 → MinHash\+LSH 去重
> 
> 



- **规则过滤**：用正则等硬性规则扔掉明显不合格的文本（非中文、乱码、过短等）。

- **KenLM 困惑度初筛**：训练一个轻量级N\-gram语言模型（KenLM）计算文本困惑度（PPL），过滤掉语言混乱、不通顺的文本。优势是**速度极快**，适合对海量数据进行第一轮粗筛。

- **MinHash\+LSH 去重**：MinHash将文本压缩成固定长度的签名，LSH将这些签名分桶，从而快速找出Jaccard相似度高的文本对，实现近似去重。相比精确去重，它能识别改写后的重复内容。

    

**面试必问点**：

- **“为什么用KenLM而不是GPT\-2等小型Transformer？”**

    - 答：速度。KenLM是传统N\-gram模型，CPU上每秒能处理几十万句，作为初筛效率极高。我们只需要它过滤掉明显的“垃圾文本”，精细的质量评估放在后面。

- **“MinHash\+LSH的原理？Jaccard相似度怎么算？”**

    - 答：通过多个哈希函数把文本的n\-gram集合映射成一组最小哈希值（签名），两个文本的相似度近似等于它们签名一致的比例。LSH通过分桶，只比较可能相似的文本，避免O\(n²\)比对。

- **“这个流水线处理过最大的数据量是多少？遇到过什么挑战？”**

    - 可以结合项目回答：例如“候选池67k时这套流程完全足够，因为轻量，内存占用很小。如果数据量再大两个数量级，我会把LSH参数调到更稀疏的桶，或者分片处理。”

#### 模块2：指令合成与偏好数据构造

**定位**：飞轮的“数据生产车间”，生成多样化的指令和对齐所需的偏好数据。



**内容拆解**：

1. **指令合成（Self\-Instruct / Evol\-Instruct）**

    - **Self\-Instruct**：给定少量种子任务，让一个“老师模型”自己生成新的指令和对应的回答。核心是一个Prompt模板，告诉模型“生成更多类似的任务”。

    - **Evol\-Instruct**：在Self\-Instruct的基础上，要求模型自动把简单指令“进化”成更复杂的版本（如增加约束、步骤、推理深度），提高指令多样性。

2. **DPO偏好数据自动构造（规则驱动 \+ 奖励模型二次过滤）**

    - 简历中写：“设计规则驱动的DPO偏好数据自动构造流程（代码执行正确性、数学答案验证等），结合奖励模型二次过滤”。

    - 这意味着你**不用人工标注**，而是：

        - 对于代码任务：通过执行单元测试，生成“正确/错误”的答案对作为 chosen/rejected。

        - 对于数学任务：通过比对最终答案（数值/表达式），生成偏好对。

        - 再拿一个已有的奖励模型对这些偏好对打分，过滤掉那些“chosen回答并不明显好于rejected”的噪声对，保证偏好信号纯净。

            

**面试必问点**：

- **“Evol\-Instruct具体怎么进化指令？举个例子。”**

    - 答：比如原始指令是“写一个快速排序”，进化后可能变成“用Python写一个快速排序，要求添加详细的注释，并分析其平均时间复杂度和最坏情况”。

- **“你的DPO偏好数据完全是自动构造的，怎么保证质量？”**

    - 答：通过两道关口。第一道是规则（代码执行、数学答案验证），它们是客观无误的信号；第二道是奖励模型过滤，它是一个轻量级的质量哨兵，能剔除那些虽然规则正确但语义质量差（如格式混乱）的pair。

- **“为什么不用纯粹的LLM排序来构造偏好对？考虑过成本吗？”**

    - 答：可以承认考虑过，但我们的自动规则\+RM方式成本接近于0，且数学/代码领域的规则是绝对可靠的；而LLM排序可能引入偏好错误，并且调API有成本。

#### 模块3：数据质量评估体系

**定位**：飞轮的“质检中心”，对合成后的数据打分，指导后续的课程学习，并作为数据迭代的决策依据。



**内容拆解**（简历原文）：

> PPL 流畅度 \+ IFD 指令难度 \+ Embedding 多样性
> 
> 



- **PPL 流畅度**：同样用KenLM或小模型计算生成文本的困惑度，评估单条数据的语言流畅性。

- **IFD 指令难度**（Instruction\-Following Difficulty）：这是一个非常关键的面试亮点。

    - 原理：让一个已经微调过的小模型，分别计算 **“给定指令时生成答案的PPL”** 和 **“不看指令，直接生成答案的PPL”**。指令带来的PPL降低幅度越大，说明指令对模型的引导作用越强，这个指令就越“简单”（因为答案更确定）；降低得越少，说明指令本身很难提供有效引导，属于“困难”任务。

    - 你在这里可以进一步解释：我们用这个分数作为课程学习的排序依据，让模型先学简单的，再学难的。

- **Embedding 多样性**：把所有的指令用预训练句子编码模型（如`all-MiniLM-L6-v2`）编码成向量，计算整体数据集的余弦相似度分布。相似度均值越低，说明数据多样性越好。这是一个**宏观监控指标**，用来防止合成数据多样性坍塌。

    

**面试必问点**：

- **“IFD具体怎么计算的？公式是什么？”**

    - 要能写出来：IFD ≈ PPL\(answer \| instruction\) / PPL\(answer\)。通常用模型计算loss来近似PPL。较低的IFD值代表简单指令。

- **“这个三维评估体系是如何协同工作的？”**

    - 答：PPL是基础流畅性门槛，过滤掉不通顺的；IFD是难度分级器，服务于课程学习；Embedding多样性是全局监控，如果发现多样性下降，就触发数据合成策略调整（比如提高生成温度）。

- **“如果一个数据IFD分数很低，但PPL很高，你怎么看？”**

    - 这可能是一段很流畅但完全无关的回复，说明指令很简单但答案生成错了。这种数据会被PPL门槛筛掉。三维指标要同时满足。

#### 模块4：课程学习排序与模型训练

**定位**：飞轮的“训练策略”，用排序后的数据以从易到难的方式训练模型，提升学习效率和最终性能。



**内容拆解**：

- 基于上面的IFD分数，将训练数据从“最容易”到“最困难”排序。

- 在训练过程中，不是随机shuffle，而是按这个顺序喂给模型。通常先训练容易的epoch，再逐步引入困难数据。

    

**面试必问点**：

- **“课程学习为什么有效？你这个项目里验证过吗？”**

    - 答：类似于人类学习，从简单到难可以避免模型初期被极难样本弄懵，稳定收敛。我们通过对照实验验证了（虽然可能没有专门写进简历，但可以提到小规模实验证明loss下降更稳定）。

        

---



#### 模块5：下游评估与反馈闭环（核心亮点）

**定位**：整个飞轮的“方向盘”，让数据工程不再是单向的，而是能根据模型表现自我优化。



**内容拆解**：

- 对每一轮训练好的模型在评测集上打分（自动评测总分、loss等）。

- **反向驱动合成策略参数调整**：如果评测发现模型在“推理类”任务上准确率低，则调整Self\-Instruct prompt，加大推理类指令的**采样比例**，或者提高生成**温度**以增加推理指令的多样性。

- **反向驱动过滤阈值优化**：如果发现整体数据噪声变大，就收紧KenLM的PPL过滤阈值，提高准入门槛。

    

**面试必问点**：

- **“这个反馈闭环是完全自动的吗？还是需要人工介入？”**

    - 务必诚实且加分地回答：“在小规模项目里是半自动的。评估结果是自动产生的，但策略调整（比如修改Prompt模板、调整阈值）是我根据评估结果进行分析后手动操作的。这让我能深入理解每项参数的影响，为未来完全自动化积累规则。”

- **“你能不能描述一次完整的闭环迭代？”**

    - 准备好一个具体案例：例如“模型在多轮对话上表现不佳，我发现合成的对话数据很多是单轮。于是我在Evol\-Instruct的prompt里加入了‘将以下单轮任务改写为包含3轮以上交互的多轮对话任务’的指令，重新合成后，多轮能力提升了X%。”

|**课程学习**|从易到难训练，提升收敛稳定性|“你做过对比实验吗？有证据表明课程学习比随机shuffle更好吗？”|
|**反馈闭环**|让数据策略随模型表现迭代，自进化|“描述一次失败的数据迭代，你是怎么发现并修复的？”|
