## 三、阶段1 当前边界

> 边界 = 当前系统能做什么、还不能做什么。面试时被问"能上线吗"，直接回答这些。

### 已验证（可以自信说的）

- [x] 三阶段清洗管线在 C4 3679条网页语料上跑通
- [x] 规则过滤：5条规则链，non_chinese 是最活跃过滤器（C4网页特征）
- [x] N-gram PPL：手写 trigram 回退模型，PPL 与流畅度正相关
- [x] MinHash+LSH：通过 datasketch 实现近似去重，Jaccard ≥ 0.60
- [x] 2×2 网格搜索验证了参数方向性影响
- [x] 健康度对比：长度 std -70%，Bigram多样性 +36%

### 已知局限（需要诚实说的）

- [ ] **N-gram 模型用待清洗数据自训练**：当前用 3114 条待评估文本训练 N-gram 模型，再对同一批数据打分。导致 PPL 值整体偏低（median=5.2），区分力不足。生产环境应该用外部干净语料（如维基百科）独立训练 N-gram 哨兵模型
- [ ] **样本量过小**：3679 条仅验证方向性影响，非全局最优参数
- [ ] **缺少人工抽样验证**：被清洗掉的 597 条没有经过人工确认确实是垃圾
- [ ] **缺少下游训练对比**：清洗前 vs 清洗后数据分别训练同一个模型，对比 loss/评测分（需 GPU，阶段6补充）
- [ ] **单机串行**：当前全流程在单机 CPU 上串行运行。百万级数据需要 Spark/Ray 分片
- [ ] **规则过滤无白名单机制**：短法条、英文专有名词等可能被误杀
- [ ] **去重质量评分维度单一**：未利用数据来源（URL/域名权威性）等信号
- [ ] **领域参数需要重校准**：当前阈值只适用于 C4 网页语料。换法律/金融/代码数据时，PPL阈值和Jaccard阈值都需要基于新数据的分布重新校准

## 四、阶段2 当前边界

> 阶段2通过 Self-Instruct + Evol-Instruct + DPO 三模块完成指令数据自动合成。

### 已验证（可以自信说的）

- [x] Self-Instruct：80条种子 → 375条有效指令，类型分布均匀（83:80:73:72:63），总费用 ¥0.50
- [x] Evol-Instruct：5种进化算子，150候选 → 130条成功进化，成功率87%
- [x] DPO自动偏好构造：数学答案验证成功率 95%，是最可靠的无监督偏好信号
- [x] LLM-as-Judge 兜底方案：解决代码执行判定成功率仅 5% 的问题
- [x] 断点续跑 + 代理支持：handle VPN 断开、环境变量 http_proxy 配置
- [x] 四层质量验证体系：规则过滤 → 模型打分 → 抽样人工 → 下游训练
- [x] 总费用 ¥2.20，验证了低成本自动化合成的可行性

### 已知局限（需要诚实说的）

- [ ] **单 API 依赖**：当前只用 DeepSeek，不同模型合成的指令分布可能不同。生产环境应对比 2-3 家 API，或自部署开源模型
- [ ] **Self-Instruct 的模板化风险**：尽管 task_type 随机指定保障了类型多样性，但 375 条 SI 指令中同一 task_type 的内部模式可能趋同（如"代码编写"类大多要求 Python）
- [ ] **DPO 代码执行几乎无效**：5% 成功率说明"让 LLM 生成可执行正确 vs 错误代码对"在当前 prompt 下非常困难，全靠 LLM-as-Judge 兜底
- [ ] **LLM-as-Judge 主观偏差**：v4pro 对"哪个回答更好"的判断有自身偏好（偏向长度、格式规范性），引入系统偏差
- [ ] **种子数较少**：80 条种子无法覆盖所有领域，如医学、法律等专业领域指令缺失
- [ ] **后过滤一致性抽样太小**：30条一致性检查不足以检测 505 条中所有有歧义的 instruction
- [ ] **数据量级仍然偏小**：505条指令远不够训练一个实用模型，仅验证流程可行性

### 踩坑记录

| 坑 | 根因 | 教训 |
|----|------|------|
| VPN 断开后 API 全失败 | 网络连接丢失 | 加断点续跑 + 环境变量 http_proxy 支持 |
| DPO 代码执行判定几乎全 PASS | LLM 难生成"刚好执行 crash"的代码 | LLM-as-Judge (v4pro) 兜底 |
| LLM-Judge 解析失败 | max_tokens=10 被 v4pro 推理链吃光 | Judge 类调用 max_tokens ≥ 500 |
| LLM-Judge 输出格式不统一 | v4pro 自由输出 "A"/"回答A"/"A。" | 兼容多种格式解析 |

## 六、各阶段依赖关系

```
阶段1（清洗）──→ 阶段2（合成）──→ 阶段3（评估）──→ 阶段4（排序）
                                                    │
阶段5（闭环）←── 需要阶段6（训练+评测）的输出         │
    │                                               │
    └──────────────────── 反向驱动 ──────────────────┘
```

- 阶段1~4 可在本地串行完成（数据互相依赖）
- 阶段5 是纯逻辑代码，不需要等训练完成就能写（接口预留 + 硬编码模拟信号）
- 阶段6 需要 GPU 服务器，规模化验证 + 下游 loss 对比
