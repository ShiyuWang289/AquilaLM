# 第四层：规模化分布式处理方案设计

> 定位：当面试官问"数据量两个数量级怎么办"时，能画出架构图并讲清每层的改造方案。

---

## 一、当前架构的瓶颈分析

AquilaLM 第一轮在 3679 条 C4 网页语料上跑通全链路，单机 CPU + API + GPU 混合串行。当数据量增长到百万级时：

| 阶段 | 当前方式 | 瓶颈 | 百万级改造方案 |
|------|----------|------|---------------|
| 阶段1 规则过滤 | 单线程 Python for-loop | CPU 密集，O(n) | **Spark UDF 并行**，无状态规则天然可分布 |
| 阶段1 PPL | 手写 N-gram 单机 | 内存 O(vocab²)，CPU 密集 | **Ray Actor 分片**，每片独立 N-gram + 哨兵共享 |
| 阶段1 LSH | datasketch 单机 | 内存 O(n²) 候选对 | **Spark MinHashLSH**，原生分布式 LSH 分桶 |
| 阶段2 合成 | 串行 API 调用 | API rate limit，I/O 密集 | **AsyncIO + 多 endpoint 池**，AIMD 自适应并发 |
| 阶段3 IFD | GPT-2 102M 本地推理 | GPU 串行 | **Ray batch inference**，多 GPU 分片 |
| 阶段3 Diversity | 全量余弦矩阵 O(n²) | 505² = 255k 对 OK，10万² = 50亿对 → 内存炸 | **MiniBatch K-Means** 或 **随机采样**，降至 O(n) |
| 阶段6 训练 | 单 GPU SFT | VRAM 8GB | **DeepSpeed ZeRO-3** 或 **FSDP** 分片 |

## 二、数据清洗层（阶段1）的分布式方案

### 2.1 核心思想不变

```
规则过滤 → PPL 困惑度 → MinHash+LSH 去重
成本递增，每层为下一层减负

分布式只是把每层从单机 CPU 改造成多 worker 并行，逻辑不变。
```

### 2.2 架构

```
                    ┌──────────────┐
                    │   Coordinator │  (Ray Driver / Spark Driver)
                    └──────┬───────┘
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
    │  Worker 0   │ │  Worker 1   │ │  Worker N   │
    │  Chunk 0    │ │  Chunk 1    │ │  Chunk N    │
    │  ↓ 规则过滤 │ │  ↓ 规则过滤 │ │  ↓ 规则过滤 │
    │  ↓ N-gram   │ │  ↓ N-gram   │ │  ↓ N-gram   │
    │  ↓ LSH      │ │  ↓ LSH      │ │  ↓ LSH      │
    └─────────────┘ └─────────────┘ └─────────────┘
           │               │               │
           └───────────────┼───────────────┘
                           │
                    ┌──────▼──────┐
                    │  合并 + 全局去重 │
                    │  (跨分片 LSH)   │
                    └──────────────┘
```

### 2.3 关键设计决策

**为什么规则过滤用 Spark 而不是 Ray？**
规则过滤是无状态的逐条 transform——Spark 的 `mapPartitions` 原生优化最好，不需要自定义 Actor。

**为什么 PPL 用 Ray Actor？**
N-gram 模型需要维护一个内存计数表（vocab² 大小）。Ray Actor 是带状态的分布式执行单元——每个 Actor 在自己分片内独立训练 N-gram，哨兵模型全局共享。Spark 对有状态任务不够友好。

**为什么 LSH 用 Spark 的 MinHashLSH？**
因为跨分片去重需要全局桶信息——Spark MLlib 的 `MinHashLSH.approxSimilarityJoin` 原生支持这个场景。

### 2.4 面试话术

> "3679 条在单机 CPU 上串行没问题，但百万级数据量下需要分片并行。规则过滤用 Spark mapPartitions——无状态操作天然可分布。N-gram 用 Ray Actor——每个 Actor 在自己分片内训练，哨兵模型全局共享。LSH 用 Spark 原生 MinHashLSH——跨分片的近似匹配需要全局桶信息。三层递进、成本递增的逻辑不变，只是每层的执行引擎换成了分布式。"

## 三、指令合成层（阶段2）的分布式方案

### 3.1 瓶颈分析

百万级种子 × 5 条/种子 = 500 万次 API 调用。单 endpoint 串行 = 500 万 × 3s ≈ **173 天**。纯并发不行——API 有 rate limit。

### 3.2 多 Provider Async Pool

```
                    ┌────────────────────┐
                    │   Scheduler         │
                    │   (AIMD 自适应)      │
                    │   max_concurrent=动态 │
                    └──────┬─────────────┘
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
    │  DS Pool    │ │  Qwen Pool   │ │  GLM Pool   │
    │  limit:2500 │ │  limit:500   │ │  limit:200   │
    └─────────────┘ └─────────────┘ └─────────────┘
           │               │               │
           └───────────────┼───────────────┘
                           │
                    ┌──────▼──────┐
                    │  结果聚合 + 去重 │
                    └──────────────┘
```

### 3.3 AIMD 自适应并发控制

```
init:  max_concurrent = 4
loop:
  成功 → max_concurrent += 1   (试探)
  429  → max_concurrent *= 0.5  (退避)
  稳定 → 维持
```

这是 NVIDIA NeMo Data Designer 和 OmniGen 的标准做法。

### 3.4 面试话术

> "指令合成是 I/O 密集型任务——瓶颈不在算力，在 API rate limit。我把 500 万次请求分到 DeepSeek/Qwen/GLM 三个 endpoint pool，每个 pool 用 AIMD 自适应算法独立控制并发数——成功加一试探上限，遇到 429 立刻退避一半。这样既能最大化吞吐，又不会触发限频。加上 checkpoint + 增量追加，500 万次调用稳定跑完不会丢数据。"

## 四、质量评估层（阶段3）的分布式方案

### 4.1 IFD 推理

GPT-2 102M 在 A100 上 batch_size=64 推理 ~0.01s/条。10 万条约 16 秒。真正的瓶颈不是推理本身。

### 4.2 Diversity 的 O(n²) 问题

10 万条指令的全量余弦相似度矩阵 = 50 亿对，内存爆炸。

**解决方案**：随机采样 + MiniBatch K-Means：
1. 随机采样 1 万条 → 做迷你 cosine 矩阵 → 得到均值 ≈ 全量均值（中心极限定理）
2. 聚类到 100 个簇 → 簇内算多样性 → 簇间算 diversity
3. 降维到 O(n)

### 4.3 面试话术

> "10 万条的全量余弦矩阵有 50 亿对——内存和算力都不可接受。但中心极限定理让我们可以用采样近似——随机 1 万条的 cosine 均值几乎等于全量均值。加上 MiniBatch K-Means 预分桶，复杂度从 O(n²) 降到 O(n)。Diversity 是宏观监控指标，采样误差在上下 0.01 的范围内不影响判断。"

## 五、训练层（阶段6）的分布式方案

### 5.1 当前限制

RTX 4060 8GB → 只能装 104M 模型。要大模型训练必须上分布式。

### 5.2 DeepSpeed ZeRO-3

```
传统数据并行：每 GPU 存完整模型副本（浪费）
ZeRO-3：优化器状态 + 梯度 + 参数 全分片

单 GPU 需要 VRAM：
  7B 模型：7B × 2(fp16) × 3(optimizer) ≈ 42GB  ← 传统
  7B 模型：42GB / 8 GPUs ≈ 5.3GB/GPU         ← ZeRO-3
```

### 5.3 面试话术

> "RTX 4060 够让 104M 模型跑通全验证流程。但路线图里预留了 DeepSpeed ZeRO-3 的接口——参数、梯度、优化器状态全分片到多 GPU，7B 模型每卡只需 5.3GB。104M 是地面的验证，7B 是上层的缩放。"
