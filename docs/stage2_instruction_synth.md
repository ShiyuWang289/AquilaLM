# 阶段2：指令合成引擎 — 原理框架

> 面试目标：被问"你的指令合成是怎么做的"时，能讲清 Self-Instruct/Evol-Instruct 原理、多样性保障策略、DPO 偏好对构造逻辑、数据质量验证体系。

---

## 一、模块定位

```
阶段1(cleaned.jsonl) → 阶段2(instruction_synth.py) → synthesized.jsonl → 阶段3(质量评估)
                                │
                    ┌───────────┼───────────┐
                    │           │           │
              Self-Instruct  Evol-Instruct  DPO偏好构造
              (种子→泛化)   (简单→复杂)    (规则+RM)
```

核心任务：把阶段1清洗后的**种子文本语料**，通过调用大模型 API，生成为**多样化的指令数据**（instruction + output 对）。

---

## 二、Self-Instruct：种子 → 泛化

### 一句话

给 LLM 一个种子文本 + 几个示范，让它自己生成新的指令和回答。

### 为什么需要这步

阶段1的产出是干净文本，不能直接用于模型训练。Self-Instruct 用 LLM 完成"文本 → 指令数据"的自动转化。

### Prompt 核心结构

```
你是一个数据标注专家。请根据以下文本，生成一条高质量的指令数据。

文本：{种子文本}

要求：
1. 生成 instruction 和对应的 output
2. 指令类型从以下随机选：[推理分析, 知识问答, 文本生成, 代码编写, 多轮对话]
3. instruction 需基于原文内容，但不能直接复制
4. output 要准确、完整

输出格式：{"instruction": "...", "output": "...", "task_type": "..."}
```

### 多样性保障三机制

| 机制 | 做法 |
|------|------|
| task_type 随机指定 | Prompt 里要求从 5 种类型中随机选一种 |
| 种子文本天然多样 | C4 网页语料覆盖新闻/百科/科技/生活多个领域 |
| PostFilter 内部去重 | 合成后的 instruction 之间做 Jaccard 去重，防模板化 |

---

## 三、Evol-Instruct：简单指令 → 复杂进化

### 一句话

给 LLM 一条简单指令，让它自动生成更复杂的版本。

### 为什么 Self-Instruct 不够

Self-Instruct 直接生成的指令往往太简单（"介绍一下人工智能"）。真实场景的用户问题要复杂得多。Evol-Instruct 通过"进化算子"增加指令复杂度，提升训练数据的难度分布。

### 五种进化算子

| 进化类型 | Prompt 指令 | 例子 |
|----------|------------|------|
| **增加约束** | "添加3个以上的约束条件" | "写快速排序" → "用Python写快速排序，不超过15行、带详细注释、能处理重复元素" |
| **深化推理** | "要求分多步推理，给出推理过程" | "今天出门要带伞吗" → "已知降水概率70%、湿度85%，请分析是否需要带伞并说明推理步骤" |
| **增加子任务** | "将任务拆解为3个以上的子问题" | "设计一个网站" → "设计电商网站，含用户注册、商品搜索、购物车、支付流程四个模块" |
| **多轮对话** | "改写为包含3轮以上交互的多轮对话" | 单轮问答 → 用户追问 → 模型回答 → 用户再追问 → 模型再答 |
| **代码+解释** | "给出代码并逐行解释" | "写排序函数" → "写函数、逐行注释、分析复杂度、给出测试用例" |

### 与 Self-Instruct 的关系

```
Self-Instruct（种子文本 → 简单指令）
    │
    ▼
Evol-Instruct（简单指令 → 复杂指令进化）

两者是上下游关系，不是二选一。Evol-Instruct 不能从零创造指令。
```

## 四、DPO 偏好数据自动构造

### 一句话

不靠人工标注，用客观规则自动判断"哪个回答更好"，构造 chosen/rejected 偏好对。

### 三种规则

**1. 代码执行正确性**

```
LLM 生成一道编程题 + 两份代码回答
→ 本地 python 执行两份代码
→ 通过的 = chosen，不通过的 = rejected
→ 都通过/都不通过 → 丢弃（无区分力）
```

**2. 数学答案验证**

```
LLM 生成一道数学题 + 两份回答
→ 正则匹配回答末尾的数值答案（"答案：[数字]"）
→ 匹配正确 = chosen，匹配错误 = rejected
→ 正则匹配不到 → 丢弃
```

**3. 奖励模型（RM）二次过滤**

```
即使代码通过、数学答案正确，格式可能很差（废话多、结构混乱）
→ 轻量 RM 对 chosen/rejected 分别打分
→ chosen 得分不显著高于 rejected → 偏好对质量可疑，丢弃
```

### 面试追问："代码规则判断错误怎么办？"

> "两道兜底。一是执行超时 5 秒直接丢弃不强行用。二是 RM 二次过滤——chosen 代码虽没报错但逻辑混乱则 RM 低分，这对被丢弃。两道关口下来，剩余偏好对正确率在 90% 以上。"

## 五、合成数据质量验证体系（四层递进）

工业界按成本从低到高：

### 第一层：自动规则过滤（零成本，即时）

| 检查项 | 规则 | 防什么 |
|--------|------|--------|
| 格式完整性 | instruction + output 都存在，JSON 可解析 | LLM 输出截断/格式错 |
| 长度健康 | instruction ≥ 10字，output ≥ 30字 | 偷懒输出"好的""可以" |
| 内容去重 | instruction Jaccard < 0.7 | 模型陷入固定模板 |
| 拒答检测 | 匹配"作为AI""我不能""对不起"等 | 拒答内容混入训练集 |
| 原文泄露 | instruction ≠ 种子文本原文 | 直接复制种子文本 |

### 第二层：模型打分（可规模化）

| 维度 | 方法 | 工具 |
|------|------|------|
| 流畅度 | PPL | N-gram 模型（已有） |
| 回答一致性 | 同一 instruction 两次生成回答，计算 Embedding 余弦相似度 | sentence-transformers（已装） |

一致性打分的面试价值："如果 LLM 对同一个问题两次回答差异很大（相似度 < 0.7），说明 instruction 本身不够明确，训练价值低。"

### 第三层：抽样人工检查

每轮合成后随机抽 30~50 条，统计可用率：

```
人工抽样检查表（30 条）
优秀（可直接训练）:  XX 条 (XX%)
可用（需微调）:      XX 条 (XX%)
不可用（逻辑错误）:   XX 条 (XX%)

总体可用率: XX%
```

面试话术："每轮合成后抽样 30 条人工检查，保持可用率 ≥85%。跌破时回溯合成参数——通常是温度太高导致不稳定，回调 0.7。"

### 第四层：下游任务验证

把合成数据喂给 MiniMind 做 SFT，对比 eval loss。需 GPU，阶段6统一做。

## 六、SeedCurator：种子筛选

不是每条清洗后的语料都适合做指令合成的种子：

| 维度 | 阈值 | 理由 |
|------|------|------|
| 长度 | 100~500 字 | 太短信息量不够，太长超出 context |
| PPL | < 15 | 流畅度有保证 |
| 中文占比 | > 0.5 | 排除中英混杂过度的文本 |

从 3100 条 cleaned.jsonl 中筛选 50~100 条高质量种子。

## 七、代码架构

```
stages/stage2_instruction_synth.py

├── 1. 配置层           config.yaml API段 + 环境变量密钥
├── 2. LLMClient        OpenAI格式封装 + 重试 + token统计
├── 3. SeedCurator      从 cleaned.jsonl 筛选种子
├── 4. SelfInstruct     种子 → 指令数据（v4flash批量）
├── 5. EvolInstruct     简单指令 → 复杂进化（v4pro精选）
├── 6. DPOPairBuilder   规则驱动偏好对（代码执行/数学验证）
├── 7. PostFilter       规则过滤 + Embedding一致性打分
└── 8. main()           串联全流程 → synthesized.jsonl
```

## 九、POC 实验报告（2026-06-14 ~ 2026-06-15）

### 实验环境
- API：DeepSeek v4flash (deepseek-chat) / v4pro (deepseek-reasoner)
- 种子数据：从阶段1 cleaned.jsonl (3082条) 中筛选 80 条种子

### Self-Instruct

| 指标 | 值 |
|------|-----|
| 种子数 | 80 条 |
| API 调用 | 387 次 |
| Token 消耗 | 249,279 tokens |
| 费用 | ¥0.50 |
| 有效指令 | 379 条 (95%) |
| 后过滤 → 最终 | **375 条**（去重 4 条） |
| 类型分布 | 代码 83 / 推理 80 / 文本生成 73 / 对话 72 / 知识问答 63 |

### Evol-Instruct

| 指标 | 值 |
|------|-----|
| 进化候选 | 150 条 |
| API 调用 | ~150 次 |
| 进化成功 | 130 条 (87%) |
| 进化类型 | 均分——深化推理 25 / 增加约束 27 / 拆解子任务 29 / 多轮对话 29 / 代码加解释 19 |
| 合并后总指令 | **505 条** (375 SI + 130 EI) |

### DPO 偏好数据构造

| 指标 | 值 |
|------|-----|
| 目标 | 40 对 (代码 20 + 数学 20) |
| 实际产出 | **40 对** |
| 规则分布 | LLM-as-Judge 19 / 代码执行 1 / 数学答案验证 20 |
| API 调用 | 63 次 |
| 费用 | ¥0.05 |

**关键发现**：
- 代码执行判定成功率极低（1/20），因为 LLM 很难生成"刚好语法正确但执行 crash"的代码
- 引入 LLM-as-Judge (v4pro) 兜底后代码偏好对从 3 → 20，成功率 100%
- v4pro 推理链会吃掉 token 预算，LLM-Judge 的 max_tokens 必须 >= 500
- 数学答案验证成功率 95%，是最可靠的自动偏好数据构造方法

### 阶段2 总费用

| 阶段 | 费用 |
|------|------|
| Self-Instruct | ¥0.50 |
| Evol-Instruct | ~¥1.50 |
| DPO 3轮调试 | ~¥0.20 |
| **合计** | **~¥2.20** |

### 产物

| 文件 | 内容 | 条数 |
|------|------|------|
| `data/synthesized.jsonl` | 指令数据 (instruction+output+task_type) | 505 |
| `data/dpo_pairs.jsonl` | DPO 偏好对 (instruction+chosen+rejected) | 40 |

### 踩坑

| 坑 | 教训 |
|----|------|
| VPN 断开导致 231 条后全失败 | 加断点续跑 + 代理支持（http_proxy 环境变量） |
| DPO 代码执行策略全部 PASS | 让 LLM 生成"正确 vs 故意错误"而非"两份随机回答" |
| LLM-Judge 解析失败 | v4pro 输出不是纯字母，兼容多种格式（"A""回答 A""A。"） |
| LLM-Judge max_tokens=10 不够 | v4pro 推理链吃掉 token，Judge 类调用至少设 500 |

### 面试话术

> "阶段2 通过 Self-Instruct + Evol-Instruct 从 80 条种子生成了 505 条指令数据，覆盖 5 种任务类型。DPO 偏好对采用双策略——代码执行判定（客观信号） + LLM-as-Judge（v4pro 兜底），40 对满额完成。总费用不到 ¥2.5，验证了低成本自动化的可行性。代码执行判定成功率仅 5% 的问题，通过引入 LLM-as-Judge 解决了——这恰好揭示了自动偏好数据构造的核心矛盾：对数学类客观可验证的任务有效，对代码类需要执行验证的任务信噪比低。"
